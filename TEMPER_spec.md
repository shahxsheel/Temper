## What TEMPER Is

TEMPER is an environment-level evaluation and auto-remediation system for Claude deployments. It tests not the raw model — that is what model cards already do — but the entire package wrapped around it: the system prompt, skill files, tool definitions, and harness configuration. It identifies where that environment is helping or hurting model performance, generates targeted fixes, re-evaluates to confirm improvement, and honestly flags what it cannot fix.

The core insight: **the same model can perform dramatically differently across different environments. No existing tool measures this. TEMPER does.**

---

## What TEMPER Is Not

- Not a replacement for model cards or standard benchmarks (GPQA, MMLU, SWE-bench). Those test the model. TEMPER tests the environment.
- Not a prompt optimizer. It does not iterate on prompts blindly. It diagnoses specific failure modes and writes targeted artifacts to address them.
- Not a model training tool. TEMPER does not touch weights.
- Not general-purpose. For the hackathon build it is Claude-specific. The architecture is model-agnostic by design and can be extended later.
- Not a passive reporting tool. It acts on what it finds.

## The Core Architecture

```
[Claude Code — User's Machine]
│
│  User runs: @eval
│
▼
[EVAL_GUIDE.md]
   Claude Code reads this at session start.
   Instructs the agent to:
     - Collect the environment bundle
     - Ship it to the eval runner
     - Render the returning report
     - Expose the @patch command
│
▼
[Bundle Collector — Claude Code]
   Scrapes and packages:
     - system_prompt.md
     - skills/*.md
     - Tool definitions (JSON/YAML)
     - Optional: sample past conversation turns
   Outputs: structured environment_bundle.json
│
│  Ships bundle to Antigravity via Interactions API
│  (Fallback: ships directly to eval runner via API calls)
▼
[Antigravity — Isolated Cloud Environment (Google)]
   Has internal AGENTS.md defining eval behavior.
   Powered by Gemini 3.5 as judge.

   ── Phase 1: Baseline Run ──────────────────────────
   Calls Claude API with NO harness loaded.
   Runs eval suite across all 6 dimensions.
   Records baseline scores.

   ── Phase 2: Full Bundle Run ───────────────────────
   Loads user's full environment bundle.
   Runs same eval suite across all 6 dimensions.
   Records bundle scores.

   ── Phase 3: Gemini Judges Both Runs ──────────────
   Scores each dimension 0–100 for both phases.
   Computes delta: bundle score minus baseline score.
   Positive delta = environment is helping.
   Negative delta = environment is hurting.
   Flags top gaps and root cause per dimension.

   Returns: structured eval_report.json
│
▼
[Report Renderer — Claude Code]
   Presents to user:
     - Score per dimension (baseline / bundle / delta)
     - Top identified gaps with root cause diagnosis
     - Which dimensions are fixable vs structural model limits
│
│  User runs: @patch
▼
[Skill Generator — Claude Code]
   For each identified gap, generates the correct artifact type:
     - Instruction Adherence gap → system prompt patch
     - Tool Call Accuracy gap → corrected tool definition
     - Output Format Compliance gap → SKILL.md with format templates
     - Skill Trigger Precision gap → rewrites skill activation conditions
     - Latency Delta gap → SKILL.md that trims context / restructures prompts
     - Error Recovery gap → SKILL.md with explicit recovery path definitions
   Writes artifacts directly into user's project.
│
▼
[Re-eval Loop — Antigravity / Direct API]
   Re-runs eval on patched dimensions ONLY (not full suite).
   If score improves above threshold: confirms fix, marks dimension resolved.
   If score does not improve: flags as STRUCTURAL LIMITATION.
   Surfaces explanation: "This is a model ceiling. No skill can fix this."
```

## The Six Eval Dimensions

Each dimension is evaluated by Gemini-as-judge with dynamic test cases generated by reading the environment bundle. Test case count is determined by confidence signal, not quota — Gemini generates cases until it has confident signal on the dimension.

### 1. Instruction Adherence
**What it tests:** Does the model follow the constraints, rules, and behavioral specifications defined in the system prompt and skill files?
**How it tests:** Gemini reads the system prompt, extracts its constraints explicitly, then generates adversarial probes designed to surface violations of *those specific rules* — not generic instruction following.
**What fixes it:** System prompt patch — clarifying ambiguous constraints, resolving contradictions, strengthening weak specifications.

### 2. Tool / Function Call Accuracy
**What it tests:** Are defined tools being called with the correct parameters at the correct times? Are there hallucinated parameters, wrong tool selections, or missed invocations?
**How it tests:** Gemini generates tasks that require the specific tools *in the user's bundle*, then evaluates call correctness against the defined schemas.
**What fixes it:** Corrected tool definitions with explicit parameter constraints + a SKILL.md with usage examples and call patterns.

### 3. Output Format Compliance
**What it tests:** Is the model producing outputs in the structure the harness requires — JSON schema adherence, markdown formatting, field names, response templates?
**How it tests:** Gemini generates tasks that should trigger structured output, then evaluates against the format specs in the bundle.
**What fixes it:** SKILL.md with explicit format templates and positive/negative output examples.

### 4. Skill Trigger Precision
**What it tests:** Are defined skills being invoked at the right times? This covers both false negatives (skill not triggered when it should be) and false positives (skill triggered when it shouldn't be).
**How it tests:** Gemini generates scenarios that should and should not trigger each skill, evaluates invocation accuracy.
**What fixes it:** Rewrites of skill activation conditions — tightening or broadening trigger definitions as needed.
**Note:** This dimension is unique to environment-level testing. No standard benchmark measures it.

### 5. Latency Delta
**What it tests:** Is the harness adding meaningful overhead? Compares TTFT and tokens/sec between baseline (no harness) and full bundle run.
**How it tests:** Measured directly during Phase 1 and Phase 2 runs. No LLM judgment needed — raw timing data.
**What fixes it:** SKILL.md that trims unnecessary context, restructures prompt assembly, or flags runaway token usage patterns.

### 6. Error Recovery Rate
**What it tests:** When the model produces a malformed output or hits a failure case, does the harness help it self-correct, or does it compound the error?
**How it tests:** Gemini injects failures calibrated to the most likely failure modes in the specific harness, evaluates whether the environment enables recovery.
**What fixes it:** SKILL.md with explicit recovery path definitions and fallback behavior specifications.

## The Delta Column

The delta (bundle score minus baseline score) is the unique output of TEMPER that no standard benchmark produces. It answers:

- **Positive delta:** Your environment is making the model better on this dimension by X points.
- **Negative delta:** Your environment is actively making the model worse on this dimension by X points.
- **Near-zero delta:** Your environment is neither helping nor hurting here.

A model can score 90 on a standard benchmark and have a negative delta on Tool Call Accuracy — meaning the harness is degrading a capability the model natively has. TEMPER surfaces this. Nothing else does.

## Remediation Logic

Remediation artifact type is determined by failure mode, not by a fixed rule:

| Dimension | Primary Fix | Secondary Fix |
|---|---|---|
| Instruction Adherence | System prompt patch | — |
| Tool Call Accuracy | Tool definition correction | SKILL.md with usage patterns |
| Output Format Compliance | SKILL.md with format templates | — |
| Skill Trigger Precision | Skill activation condition rewrite | — |
| Latency Delta | SKILL.md with context trimming | — |
| Error Recovery Rate | SKILL.md with recovery paths | System prompt patch |

---

## The Re-eval Loop

TEMPER does not trust its own patches. After `@patch` writes artifacts:

1. Re-eval fires on patched dimensions only — not a full suite re-run.
2. If score improves above threshold (configurable, default: +20 points): dimension marked **RESOLVED**.
3. If score does not improve: dimension flagged as **STRUCTURAL LIMITATION**.
4. Structural limitations surface with an explanation: the specific reason no skill-level fix can address the gap, and what would actually fix it (model upgrade, architectural change, fundamental harness redesign).

The honest failure case is a feature, not a bug. A system that knows what it cannot fix is more trustworthy than one that patches everything silently.

---

## Cross-Model Evaluation Rationale

TEMPER uses Gemini as the judge evaluating Claude's environment. This is an architectural decision, not a prize-chasing one:

**A model should not grade its own homework.** Claude evaluating its own skill files, system prompt, and tool definitions introduces self-serving bias — the model is incentivized (by training) to rate its own outputs favorably. Gemini has no stake in Claude's environment quality. Its judgment is structurally more independent.

This also enables the most technically interesting moment in the demo: showing the score Gemini assigns vs. what the raw model would self-report.

## Demo Path (3 Minutes)

**Pre-demo setup:**
- Load the villain environment: a real Claude Code project with a deliberately broken system prompt (contradictory instructions), an incorrectly specified tool definition, and a skill with ambiguous trigger conditions.
- Pre-run the full eval. Cache eval_report.json with real scores.
- Know your numbers cold: which dimensions failed, what scores they got, what the patches will generate, what the re-eval scores will be.
- The only live computation in the demo is the re-eval. Everything else loads from cache.

**0:00 – 0:20 | The problem**
Show the villain environment on screen — a real Claude Code session with visible system prompt, skill files, tool definitions. Deliver exactly one line: *"This is a production Claude environment. The model card says the model is performing well. TEMPER disagrees."* Run `@eval`.

**0:20 – 0:50 | The report loads**
Pre-cached report renders. Six dimensions. Scores. The delta column. Pause on two numbers: *"Tool Call Accuracy: 31. Instruction Adherence: 44. This harness is actively making the model worse on both."* Let it land. Do not explain the architecture yet.

**0:50 – 1:20 | One gap in detail**
Drill into Tool Call Accuracy. Show the specific test cases Gemini ran, the failure pattern, the diagnosis. Deliver: *"The tool definition doesn't specify parameter constraints. **1:20 – 1:40 | The patch**
Run `@patch`. Watch artifacts appear in real time in the project tree — a corrected tool definition and a new SKILL.md with invocation patterns. Say nothing while this happens.

**1:40 – 2:20 | Live re-eval**
Re-eval fires on Tool Call Accuracy and Instruction Adherence only. While it runs: *"TEMPER doesn't trust its own patches. It re-evaluates every dimension it touched. If the score doesn't move, it tells you it's a structural model limitation — not a fixable environment problem."* Scores return: 31 → 79. 44 → 82.

**2:20 – 2:45 | The honest failure case**
Show one dimension that did not improve after patching — Error Recovery Rate stayed flat. TEMPER flagged it: *"This is a model ceiling. No skill can fix this."* This moment of honesty is the differentiator.

*"Your model card tests the model. TEMPER tests everything around it — and fixes what it can."*

## The Load-Bearing Component

The pre-cached eval_report.json rendering correctly with the right numbers. Every sentence in the demo references something on that report. If it fails to render, the demo dies at second 20 with no pivot available.

**Mitigation:** The report renderer reads local JSON, not a live API call. Pre-generate the report before the demo slot, validate every number, commit it to the repo. The renderer must work offline. This is non-negotiable.

## Cut Order (If Time Pressure Hits)

Do not make this decision under stress at 3am. The order is pre-committed:

1. **First cut — Dynamic test case generation.** Replace with semi-dynamic: Gemini reads the bundle and selects from a strong pre-written test bank per dimension. Demo is indistinguishable to a judge.
2. **Second cut — Antigravity.** Run eval via direct API calls from Claude Code. Cross-model architecture, delta, scores, and skill generation all survive intact.
3. **Third cut — Live re-eval.** Pre-cache post-patch scores alongside the pre-patch report. Demo narrative holds. You lose the live computation moment only.

**Never cut:** The rendered report with real scores. The skill writer producing real artifacts. The honest failure case flagging structural limitations. These three are the product.
