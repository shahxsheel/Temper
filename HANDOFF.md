# TEMPER — Handoff Document

## Goal

Build **TEMPER**: an environment-level evaluation and auto-remediation system for AI deployments, for the **AI Engineer World's Fair Hackathon 2026**. It tests the harness wrapped around a model (system prompt, skills, tool defs) rather than the model itself, computes a delta vs. a bare baseline, and writes targeted fixes.

Primary prize target: **Best Usage of Gemini 3.5** (GCP credits, inference funded).

Read `TEMPER_spec.md` for the full architecture. Read `contract/api.md` for the API contract between the two devs.

## Team Split

- **Dev 1 (you)** — Local layer: Claude Code client, bundle collector, harness runner (DeepSeek), test loop, report renderer, @patch writer, re-eval trigger.
- **Dev 2** — Cloud layer: Antigravity/FastAPI server, Gemini test generator, bare DeepSeek baseline, Gemini judge, patch artifact generation.

## Key Architectural Facts (do not get wrong)

1. **Test-taker is DeepSeek, not Claude.** The harness run (local, Dev 1) calls DeepSeek with the full bundle. The bare baseline (cloud, Dev 2) calls DeepSeek with no harness. Same model → delta is clean.
2. **Claude Code is the inference client, not the inference model.** Claude Code drives the loop; it never calls the Claude API for answers.
3. **Patch content is generated by Gemini server-side.** Dev 1 only *writes* the artifacts returned in `/results`. Dev 1 does not generate fix content.
4. **`harness_score` not `bundle_score`** — this field name is frozen in the schema and contract.
5. **Latency Delta has no generated questions.** It is computed from `latency_ms` submitted with answers to the other dimensions' questions.

## The 5 API Endpoints (frozen in `contract/api.md`)

```
POST /register          body: {bundle}                  → {session_id}
GET  /next-question     ?session_id=                    → {status, question_id, dimension, prompt}
POST /submit-answer     body: {session_id, question_id, answer, latency_ms}  → {received:true}
GET  /results           ?session_id=                    → {status, report, patches}
POST /reeval            body: {session_id, dimensions, updated_bundle}       → {reeval_session_id}
```

`/results` returns `processing` for first ~2 polls then `ready`. `/next-question` returns `not_ready` while generation is running, `question` while serving, `done` when queue is empty.
